# CLAUDE.md — fast-infra

Context for AI assistants working on this repo. Read this before making changes.

## What this is

A minimal self-hosted deploy platform for a single ~4GB VPS. Containerized apps (Go, Spring Boot, FastAPI, Express, static sites) get zero-downtime rolling deploys, automatic HTTPS via Traefik, shared Postgres/Redis, and OpenObserve for logs/traces. Images are built in GitHub Actions, pushed to GHCR, deployed over SSH with the `platform` CLI.

## Core design principle

**Fewer decisions beat more features.** The target UX is: install script → DNS → `platform new` → `git push` → app live on HTTPS in 5 minutes. Every proposed feature must pass this test: does it serve the 5-minute experience, or does it add a decision? This is what differentiates the project from Coolify/Dokploy (which expose hundreds of settings).

## Deliberate architectural decisions (do not reverse casually)

- **No daemon, no REST API, no web UI.** The only way in is SSH. A Docker-socket-connected HTTP API is root-equivalent and would make us responsible for auth/TLS/audit for every user's VPS. Revisit only in v3, only if the project gets real adoption.
- **CLI is stdlib-only Go, zero external dependencies.** It shells out to the `docker` CLI instead of using the Docker SDK (SDK pulls a huge dependency tree). Keep it readable in ten minutes.
- **Traefik over Caddy/nginx** because label-based discovery means adding an app never touches central config, and `--scale` replicas are load-balanced automatically.
- **app.yaml is a flat key:value file** parsed by hand (no YAML library). Secrets never go in app.yaml — they live in `apps/<name>/.env` (chmod 600, gitignored).
- **compose files are template-rendered** (text/template embedded in the binary), not generated by a compose-model library. Escape hatch: `manual: true` in app.yaml stops the platform from rewriting the compose file.
- **Deployment history is JSONL** (`apps/<name>/.deployments`), not SQLite — no cgo, no locking, greppable. Rollback = last entry with status "success" and a different tag.
- **Rolling deploy algorithm** (cli/deploy.go): render → pull → start new replicas alongside old (`--scale old+N --no-recreate`) → poll Docker healthchecks (120s cap) → SIGTERM old with 30s drain → settle scale → record history. A failed deploy removes only the new containers; old ones never stop serving. Traefik's Docker discovery handles traffic shifting; there is no explicit route-switch step.
- **CI pushes every image twice**: `:latest` and `:<commit-sha>`. Rollback depends on SHA tags staying in the registry.
- **OpenObserve over Grafana+Loki+Tempo** (single ~300-500MB binary vs 1.5GB+ stack). **Adminer over pgAdmin** (~25MB vs ~150MB). **Dozzle** for live log tail instead of writing a log viewer. **RabbitMQ behind a compose profile**, off by default.
- One Postgres instance, one database per app. One Redis, key-prefix-per-app (never db indexes — they don't survive a move to Redis Cluster).

## App contract

Apps must: (1) listen on `port` — the compose template injects it as the `PORT` env var, so `port` in app.yaml is the single source of truth for the health check, Traefik, and the app itself; frameworks that read a different variable (Spring's `SERVER_PORT`) must set it in `.env` to match; (2) expose an HTTP health endpoint at `health`; (3) contain `wget` or `curl` — the generated healthcheck shells out to one (scratch/distroless images have neither, so add one or use `manual: true` with your own `HEALTHCHECK`); (4) handle SIGTERM gracefully (finish in-flight requests, exit within 30s). `examples/go-hello` is the reference implementation. Java apps must set `-Xmx` (deploys briefly run two copies; budget peak, not steady state). DB migrations must be backward-compatible across one release (old+new run concurrently during the swap). `manual: true` compose files must still define a `healthcheck` — the rolling deploy polls Docker health status and times out (120s) without one.

## Layout

```
install.sh                  one-shot VPS bootstrap (generates all passwords)
infra/docker-compose.yml    shared services, network name: platform
cli/                        the `platform` binary (new/deploy/rollback/scale/status)
cli/templates/              compose template embedded via go:embed
workflows/deploy-template.yml   reusable GH Actions workflow (copy into fork's .github/workflows/)
examples/go-hello/          reference app + 10-line caller workflow
apps/                       per-app dirs on the VPS (gitignored): app.yaml, .env, compose, .current_tag, .deployments
```

## Roadmap discipline

- v1 (current): everything above.
- v2 (in progress): `platform env` list/set/unset — **done** (cli/env.go); still to do: sequential rolling for multi-replica apps, richer status.
- v3 (only with real users): REST API + web UI reusing the same Go internals; app.yaml schema growth.

Do not build v3 features during v1/v2. When in doubt, ship less.

## Conventions

- Commit messages: plain imperative English, no AI attribution or Co-Authored-By trailers.
- Shell scripts: `set -euo pipefail`, LF line endings (`*.sh text eol=lf`).
- New CLI commands: add to the switch in cli/main.go, keep files per-command, stdlib only.
- Docs live in README.md; keep it prose-first and short.
